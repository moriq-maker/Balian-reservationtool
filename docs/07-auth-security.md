# 07. 認証・セキュリティ設計書

## 1. 一般スタッフ:共通アクセスコード方式(承認済み)

個人ログインは行わず、**共通アクセスコード(PIN)**による端末単位の保護を採用する。

### 1-1. 方式概要(確定仕様)

1. 一般スタッフ画面(予約表)へのアクセス時、初回(またはセッション切れ時)に共通アクセスコード入力画面を表示する。
2. **コードは6桁の数字**とし、管理者が設定・変更できるようにする(`system_settings`に保存、値自体はハッシュ化して保存)。
3. 認証成功後、httpOnly・secure・sameSite=strict属性のセッションCookieを発行する。**有効期限は1年**とし、フロント端末に固定的にログイン状態を維持する運用とする(端末を共有PC/タブレットとして扱うため、日々の再入力は発生しない想定)。
4. コード自体をURLパラメータや平文でクライアントに保持させない(Cookieのみで判定)。

### 1-2. 補完策(推奨・任意)

- URLを推測されにくい非公開パス(例:`/staff/x7k2p9`)にすることで、コード漏洩前の第一防波堤とする(シークレットURL)。採用可否は未確定([7. 未確定事項](#7-未確定事項承認待ち)参照)。
- セッションが1年間有効なため、**万が一コードや端末が漏洩した場合に備え、管理画面から「全セッションを強制的に無効化する(コード再発行と同時に既存Cookieを無効にする)」操作**を用意する。
- アクセスコードは定期的に変更する運用ルールを運用マニュアル([10-deployment-operation.md](10-deployment-operation.md))に明記する(コードを変更しても、既にログイン済みの端末セッションはコード自体を保持していないため即座には失効しない点に注意し、上記の強制無効化操作と併用する)。
- ログイン試行回数制限(例:6桁数字は総当たり耐性が低いため、5回失敗で一定時間ロックは必須とする)を設け、総当たり攻撃を抑止する。

### 1-3. 採用しなかった候補とその理由

| 候補 | 不採用/保留の理由 |
|---|---|
| IP制限 | ホテル側のネットワーク構成(固定グローバルIPの有無)に依存し、Vercel側だけでは確実に制御できないため、補完策止まりとする |
| 端末証明書などの端末認証 | 導入・運用コストが高く、初学者が保守するには複雑すぎる |
| 個人ログイン | 要件上、一般スタッフには個人ログインを課さない方針のため不採用 |

## 2. 管理者認証

Supabase Authまたは自前実装のメール+パスワード認証を採用し、管理者専用ログイン画面からログインする。

**運用方針(承認済み)**:管理者アカウントは**1つのみ発行し、社員間で使い回す**。

- ログイン後は管理者専用のセッションCookie(一般スタッフ用とは別のCookie名)を発行する。
- `admin_users`テーブルは複数アカウントに対応できる設計のまま維持し、将来的にアカウントを分割したい場合は追加発行のみで対応できるようにする。
- **限界の明記**:共通アカウント運用である以上、「どの管理者が」操作したかはシステム上追跡できない。重要操作(施設利用停止、一括キャンセル等)を行った際は、画面上で「誰が実施したか」を口頭/運用ルールで管理することを運用マニュアルに明記する。

## 3. 画面・権限の分離

- 一般スタッフ画面と管理者画面は別ルート(例:`/staff/...` と `/admin/...`)に分離する。
- 一般スタッフ用Cookieのみでは管理者専用API(施設マスター変更・利用停止・アナリティクス・履歴検索・システム設定)を一切呼び出せないよう、Server Action/Route Handler側で必ず管理者セッションを検証する。
- 管理者は一般スタッフ画面の全操作も行えるため、管理者ログイン後は一般スタッフ向け機能+管理者向け機能の両方にアクセスできるUIとする。

## 4. セッション管理

| 対象 | Cookie名(例) | 有効期限 | 属性 |
|---|---|---|---|
| 一般スタッフ | `staff_session` | **1年(確定)** | httpOnly, secure, sameSite=strict |
| 管理者 | `admin_session` | 12時間(要調整、ログアウトまたは期限切れで再ログイン) | httpOnly, secure, sameSite=strict |

一般スタッフのセッション有効期限は1年で確定。管理者セッションの正確な時間は運用開始後に調整可能な設定値とする。

## 5. セキュリティ対策

| 項目 | 対策 |
|---|---|
| CSRF | Server Actionsの仕組み(Next.jsの組み込みCSRF保護、Originヘッダ検証)を利用。Cookieは`sameSite=strict`を基本とする |
| XSS | ユーザー入力(氏名・備考等)はエスケープして表示、リッチテキスト入力は許可しない。サーバー側でも入力をサニタイズ |
| SQLインジェクション | Prisma(パラメータ化クエリ)を使用し、生SQLを書く箇所(EXCLUDE制約関連のマイグレーション以外)を作らない |
| パスワード管理 | 管理者パスワードはbcrypt等でハッシュ化して保存。平文保存・ログ出力を禁止 |
| 環境変数管理 | DB接続文字列・セッション署名鍵・共通アクセスコードのハッシュ化用ソルト等はVercelの環境変数で管理し、リポジトリにコミットしない(`.env`は`.gitignore`対象) |
| 本番・開発分離 | Supabaseプロジェクトを本番・開発で分離し、環境変数で接続先を切替える |
| 総当たり対策 | アクセスコード・管理者パスワードともに試行回数制限を設ける |

## 6. 個人情報の取り扱い

宿泊者の部屋番号・氏名は予約管理に必要な最小限の情報のみを保存する。電話番号・メールアドレス・生年月日等の追加個人情報は収集しない。アナリティクス集計では個人が特定可能な形(氏名そのもの等)での表示は行わず、件数・比率などの集計値のみを扱う。
