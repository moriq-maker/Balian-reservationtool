# 03. 通知・自動処理設計書

## 1. 通知一覧

| 通知種別 | 発生条件 | 内容 |
|---|---|---|
| 予約開始15分前 | 予約開始15分前に到達 | 施設名、部屋番号または名前、予約開始時刻 |
| 清掃警告 | 岩盤浴が清掃中のまま、次の予約開始30分前に到達 | 施設名、次の予約開始時刻、経過時間の強調表示 |
| 施設利用停止 | 管理者が施設を利用停止に設定 | 施設名、停止期間、停止理由 |
| 新規予約登録 | 予約が新規登録された | 施設名、部屋番号または名前、開始時刻 |
| 予約変更 | 予約内容が変更された | 変更内容の概要 |
| 予約キャンセル | 予約がキャンセルされた | 施設名、部屋番号または名前、開始時刻 |

## 2. 通知UI

- 画面右上のベルアイコン + 未読件数バッジ
- 通知一覧パネル(新しい順)
- 新規発生時のトースト通知
- 既読/未読の管理

### 既読管理の方式(承認済み)

一般スタッフに個人ログインがなく、管理者アカウントも共通1つで運用するため、**既読/未読は「全スタッフ共通」でサーバー側に一元管理する**(端末ごとのローカル管理は行わない)。`notifications` テーブルに `is_read` フラグを持たせ、いずれかの端末で既読にした通知は全端末で既読として表示される。

## 3. 清掃警告の詳細仕様(改訂版)

要件当初は「予約開始時刻になっても清掃中の場合に警告」だったが、ヒアリングにより以下へ改訂する。

1. **通常警告**:岩盤浴が「清掃中」の状態で、その施設の次の予約の開始30分前に到達した時点で警告を発生させる(通知一覧・予約表上のバッジ表示の両方)。
2. **超過警告**:通常警告を出したあとも清掃中のまま予約開始時刻を過ぎた場合、警告表示を継続し、より強い視覚表現(赤色・点滅アイコンなど、詳細は [04-screen-specification.md](04-screen-specification.md))に切り替える。
3. 清掃中ステータスは新規予約の登録可否には影響しない(将来予約は登録可能)。実際に利用者を入室させてよいかは現場スタッフの判断に委ねる。

## 4. 自動処理一覧

| 処理 | 内容 | 対象 | 対象外 |
|---|---|---|---|
| 利用完了への自動変更 | 終了日時を過ぎた予約を「利用完了」に変更 | 予約中、利用中 | キャンセル |
| 岩盤浴の清掃中への自動変更 | 岩盤浴の予約が利用完了になった時点で対象施設を「清掃中」に変更 | 岩盤浴LAUT/GUNUNGのみ | - |
| 予約開始15分前通知の生成 | 開始15分前の予約を検出し通知を作成 | 予約中、利用中 | キャンセル |
| 清掃警告の生成 | 清掃中施設の次予約開始30分前を検出し警告を作成 | 岩盤浴LAUT/GUNUNGのみ | - |

いずれの処理も**同じ入力に対して何度実行しても結果が変わらない冪等な設計**とする(例:「すでに利用完了の予約を再度利用完了にしようとしても何も起きない」「同一の15分前通知を重複生成しない」)。

## 5. 自動処理の実行方式(推奨案・確定)

### 5-1. 比較検討

| 方式 | 信頼性 | 無料枠 | 保守性 | 実装難易度 | 備考 |
|---|---|---|---|---|---|
| Vercel Cron Jobs | 中(Hobbyプランは高頻度実行に制限あり) | Hobbyで利用可(実行頻度に制限) | 高い(Vercel標準機能) | 低い | 単体では分単位の精度を無料枠で出しにくい |
| Supabase Cron / pg_cron | 中〜高 | 一部プランで制限あり | 中 | 中 | Supabase側の設定が別途必要 |
| 画面アクセス時に状態を同期 | 高(営業中は画面が常時開いている前提) | 追加コスト不要 | 高い(アプリ内ロジックのみ) | 低い | 無人時間帯は同期が発生しない |
| 外部無料Cronサービス(cron-job.org等) | 中 | 無料 | 中(外部サービス依存) | 低い | 第三者サービスへの依存が増える |

### 5-2. 採用方式

ホテルフロントは営業時間中ほぼ常に画面が開いている業務特性を踏まえ、以下の二段構えを採用する。

1. **主軸:画面アクセス時同期**
   予約一覧取得など主要なAPI呼び出し(サーバー側)のたびに、以下を冪等にチェック・実行する軽量な同期処理を差し込む。
   - 終了日時を過ぎた「予約中/利用中」予約 → 「利用完了」に変更
   - 利用完了になった岩盤浴予約 → 対象施設を「清掃中」に変更
   - 開始15分前に到達した予約 → 通知未生成なら生成
   - 清掃中施設の次予約開始30分前 → 警告未生成なら生成

2. **補完:Vercel Cron(または無料の外部Cron)**
   深夜など無人の時間帯の取りこぼしに備え、数分〜数十分間隔で同一の同期処理を呼び出すエンドポイントをCronから叩く。無料枠開始の方針(承認済み)のため、まずはVercel Hobbyの範囲で構成し、実行頻度が不足する場合はcron-job.org等の無料外部Cronを補完的に併用する。Proプランへの移行要否は運用実績を見て判断する。

両経路が同じ冪等な同期関数を呼ぶため、二重実行や実行漏れがあっても状態は必ず正しい値に収束する。

```mermaid
flowchart LR
    A[スタッフが予約表を開く/操作する] --> B[同期処理を実行]
    C[Vercel Cron / 外部Cron] --> B
    B --> D[終了予約を利用完了化]
    B --> E[岩盤浴を清掃中化]
    B --> F[15分前通知を生成]
    B --> G[清掃警告(30分前)を生成]
```

## 6. リアルタイム更新 vs 定期再取得

複数端末での同時利用を前提に、以下を比較する。

| 方式 | メリット | デメリット |
|---|---|---|
| リアルタイム更新(Supabase Realtime等) | 他端末の変更が即座に反映され、二重登録の体感リスクが低い | 実装がやや複雑、コネクション管理が必要 |
| 定期再取得(ポーリング) | 実装がシンプルで理解しやすい | 反映まで数秒〜数十秒のタイムラグが生じる |

### 推奨案

Supabase Realtimeを用いたリアルタイム更新を基本採用する。予約テーブルの変更をSubscribeし、他端末での新規登録・変更・キャンセルを即時反映する。実装難易度は定期再取得よりやや高いが、Supabase無料枠内で利用可能であり、重複予約の体感的な発生を抑えられるメリットが大きい。フォールバックとして、Realtime接続が切れた場合は一定間隔(例:30秒)のポーリングに自動切替する設計とする。

## 7. 通知データの保持方針

通知は既読/未読に関わらず一定期間(例:30日)保持し、古いものは自動的にアーカイブ対象とする方針を運用時に検討する(初期実装では削除せず全件保持とし、必要に応じて後日バッチ整理を追加する)。
