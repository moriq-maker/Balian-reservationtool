## 3. メイン予約表UI案(3案)

いずれも「横軸:施設」「縦軸:時間(15分刻み)」「1日単位表示」を基本構造とし、現行スプレッドシートの考え方を踏襲します。

### 案A:スプレッドシート踏襲型グリッド

```
┌─────────┬──────────┬──────────┬────────────┬────────────┬────────┬──────────┐
│  時刻   │カラオケ101│カラオケ102│岩盤浴 LAUT │岩盤浴GUNUNG│  PJ    │ランドリー │
├─────────┼──────────┼──────────┼────────────┼────────────┼────────┼──────────┤
│ 09:00   │          │          │            │            │        │          │
│ 09:15   │ ▓301 利用中▓       │            │  【清掃中】 │        │          │
│ 09:30   │ (18:00-19:30例)    │            │            │        │          │
│ ...     │          │          │            │            │        │          │
│ 現在時刻 ─────────────────────────────────────────────────────────────      │
└─────────┴──────────┴──────────┴────────────┴────────────┴────────┴──────────┘
```

- 全施設を横並びで表示し、時間軸に沿って予約ブロックを配置。現行スプレッドシートと視覚的にほぼ同一。
- **メリット**:移行するスタッフにとって最も分かりやすい。全施設の空き状況を一画面で俯瞰できる。
- **デメリット**:施設数が増えると横幅が窮屈になる。タブレットでは横スクロールが前提になる。
- **PCでの使いやすさ**:◎(全施設横並びが可能な画面幅を確保しやすい)
- **タブレットでの使いやすさ**:△(横スクロールが必須、15分刻みの細かいセルはタップ精度が要求される)
- **スプレッドシートからの移行しやすさ**:◎(最も違和感が少ない)

### 案B:カテゴリタブ切替型

```
[カラオケ] [岩盤浴] [その他]  ← カテゴリタブで切替

┌─────────┬──────────┬──────────┐
│  時刻   │カラオケ101│カラオケ102│
├─────────┼──────────┼──────────┤
│ 09:00   │          │          │
│ ...     │          │          │
└─────────┴──────────┴──────────┘
```

- 施設カテゴリ(カラオケ/岩盤浴/その他)ごとにタブを切り替えて表示。1タブあたりの施設数は最大2〜3。
- **メリット**:1画面に表示する列数が絞られるため、タブレットでも1施設あたりの幅を広く取れ、タップ操作がしやすい。
- **デメリット**:全施設を横断した空き状況の俯瞰性が下がる(カテゴリをまたいだ比較がしづらい)。タブ切替の一手間が増える。
- **PCでの使いやすさ**:○(広い画面ではタブ自体が不要にも感じられる)
- **タブレットでの使いやすさ**:◎(横スクロール不要、タップ精度も確保しやすい)
- **スプレッドシートからの移行しやすさ**:○(全体俯瞰の感覚は変わるが、施設単位では分かりやすい)

### 案C:時間帯フォーカス型(現在時刻中心スクロール+施設横並び簡略表示)

```
[◀前日] [7月20日(月) ▼] [今日] [翌日▶]

現在時刻(14:32)を画面中央に自動スクロール

┌──────────┬──────────┬────────────┬────────────┬────┬──────────┐
│カラオケ101│カラオケ102│岩盤浴 LAUT │岩盤浴GUNUNG│ PJ │ランドリー │
│ [予約可] │ [利用中] │ [清掃中]   │ [予約可]   │... │   ...    │
│ 直近の予約ブロックのみ簡略カード表示、詳細は縦スクロールで展開 │
└──────────┴──────────┴────────────┴────────────┴────┴──────────┘
```

- 案Aの全施設横並びグリッドをベースにしつつ、初期表示時に現在時刻付近へ自動スクロールし、直近の予約状況をカード形式で見やすく強調する派生案。
- **メリット**:フロント業務中に「今」「これから」の状況を素早く把握できる。案Aの俯瞰性を保ちつつ初動の視認性を改善。
- **デメリット**:実装がやや複雑(自動スクロール制御、カード/グリッドのハイブリッド表示)。
- **PCでの使いやすさ**:◎
- **タブレットでの使いやすさ**:○(横スクロールは残るが、自動フォーカスにより目的の情報に素早く到達できる)
- **スプレッドシートからの移行しやすさ**:○(見た目は案Aに近いが、初期スクロール位置が異なる点に慣れが必要)

**採用決定**:タブレットを主端末とする運用方針のため、**案B(カテゴリタブ切替型)**を採用します。

### 3-1. 確定UI:案Bのタブレット最適化仕様

タブレットでの視認性・操作性を最優先し、以下の方針で詳細化します(フェーズ8で実装時に微調整)。

- **カテゴリタブ**:「カラオケ」「岩盤浴」「その他」の3タブを画面上部に固定表示(スクロールしても常に見える位置)。タブ自体も高さ44px以上を確保し、タップ領域を広く取る。
- **列幅**:1タブあたり最大3施設のみ表示するため、施設1列あたりの幅をPC表示時より大きく確保できる。タブレット横向き(landscape)を基本レイアウトとし、施設名・ステータスバッジ・予約ブロックの文字サイズをPC版より一段階大きくする。
- **時間軸のセル**:15分刻みの行は、タブレットの指でのタップを想定して1行あたりの高さをPC版より高めに設定し(誤タップ防止)、予約が入っている時間帯は複数行を結合したブロックとして表示する。
- **配色とコントラスト**:フロント業務は照明環境が一定でないため、予約可能(緑系)/利用中(青系)/清掃中(黄〜オレンジ系)/利用停止(グレー)のステータス色は彩度・明度差を大きく取り、色覚特性に配慮してラベル文字とアイコンを必ず併記する。
- **現在時刻ライン**:太めの線+色分けで強調し、タブ切替後も現在時刻位置へ自動スクロールする。
- **ドラッグ&ドロップ**:[11. ドラッグ&ドロップ](02-business-rules.md#11-ドラッグドロップ)の長押し方式を前提に、長押し開始時に予約ブロックを視覚的に持ち上げる(影・拡大)フィードバックを入れ、誤操作を防ぐ。
- **タブ間の空き状況把握**:全体俯瞰性が下がるという案Bのデメリットを補うため、各タブのラベル内に軽量な空き状況インジケーター(例:「岩盤浴 ●空きあり」)を表示し、タブを開かなくてもおおよその状況が分かるようにする。
- **PC表示**:同じ案Bの構造を維持しつつ、画面幅に余裕があるため2タブ分を同時表示できるレイアウトも検討する(詳細はフェーズ8で調整)。

## 4. 新規予約モーダル

入力順序(手入力を最小化する順):

1. 予約者区分(ボタン選択:宿泊中/チェックイン前/チェックアウト後)
2. 区分に応じて「部屋番号(検索可能セレクト)」または「氏名(テキスト入力)」を表示
3. 利用施設(カード or ボタン選択、色分け表示)
4. 利用開始日(カレンダーUI、当日〜設定日数先のみ選択可)
5. 利用開始時刻(15分刻みのセレクト)
6. 終了日時(自動計算・表示のみ、編集不可)
7. 備考(任意、テキストエリア)

保存ボタン押下後は二重送信防止のためボタンを無効化し、ローディング表示を出す。サーバー側で重複・利用停止チェックを行い、NG時は分かりやすい日本語エラーを表示する。

## 5. 予約詳細モーダル

表示項目:予約者区分、部屋番号または氏名、施設、開始/終了日時、予約ステータス、備考、変更履歴(時系列)。

操作:予約内容の変更、利用中への変更、キャンセル、備考の追加・編集。管理者専用操作はモーダル内に表示しない(一般スタッフ・管理者で同一モーダルを出し分ける)。

## 6. 変更確認モーダル(ドラッグ&ドロップ/施設変更)

表示項目:変更前施設、変更後施設、変更前開始/終了日時、変更後開始/終終了日時、予約者情報。

例文:「予約をカラオケ101から岩盤浴 LAUTへ変更しますか?施設の利用時間が異なるため、終了日時は自動的に再計算されます。」

重複がある場合は確定ボタンを無効化し、理由を表示する。

## 7. キャンセル確認モーダル

「この予約をキャンセルしますか?この操作は取り消せません。」+ 予約概要を表示。誤操作防止のため確定ボタンは明確に区別した配色にする。

## 8. 清掃中・利用停止の表示

- 清掃中:施設ヘッダーに「清掃中」バッジ(黄色系)。次予約開始30分前警告時はオレンジ、開始時刻超過時は赤の強調表示に切替。
- 利用停止:施設ヘッダーに「利用停止」バッジ(グレー/赤)。該当時間帯のセルはタップ不可(斜線パターン等で予約不可を明示)。

## 9. 通知一覧パネル

ベルアイコン+未読件数バッジ。パネル内は新しい順、種別ごとにアイコンを変え、未読は太字+ドットで強調。既読化は全端末共通反映(→[03-notification-jobs.md](03-notification-jobs.md))。

## 10. 履歴画面

予約詳細モーダル内:その予約に関する変更履歴を時系列カードで表示(操作種別・操作日時・操作者種別・変更前後値)。

管理者向け履歴検索画面:期間・施設・操作種別・操作者種別でフィルタ可能な一覧+詳細表示。

## 11. 管理者ログイン画面

一般スタッフ画面とは明確にURL/導線を分離する(→詳細は [07-auth-security.md](07-auth-security.md))。共通の管理者アカウント(1つ)でログインするシンプルなメールアドレス+パスワード形式を想定。

## 12. 施設管理画面

一覧(表示順・有効無効・稼働状態を一目で確認)、追加・編集フォーム(施設マスター全項目)、利用停止設定フォーム(→[02-business-rules.md](02-business-rules.md)9-2)。

## 13. アナリティクス画面

フィルターパネル+グラフ群(詳細は [08-analytics.md](08-analytics.md))。PC・タブレット横向きでの閲覧を優先したレイアウト。

## 14. PC/タブレット表示の違い

**タブレットを主端末とする方針**のため、レイアウト・文字サイズ・タップ領域はタブレット横向きを基準に設計し、PC表示はその拡張(余白活用・複数タブ同時表示など)として扱う。

| 項目 | タブレット(主) | PC(副) |
|---|---|---|
| 予約表 | 案B:カテゴリタブ切替(最大3施設/タブ)、横向き前提 | 同じ案Bベース。画面幅を活かし2タブ分の同時表示も検討 |
| モーダル | 全画面に近いサイズ、タップ操作前提でボタンを大きく | 中央配置、キーボード入力前提 |
| ドラッグ&ドロップ | 長押し→持ち上げフィードバック→移動→確認モーダル(誤操作防止) | マウスドラッグ |
| 文字・ボタンサイズ | 大きめ(タップ領域44px以上、行の高さも余裕を持たせる) | タブレットに合わせた基準をベースにやや詰めても可 |
| 配色・視認性 | ステータス色は明度・彩度差を大きく、ラベル/アイコン併記必須 | 同一方針を踏襲 |
